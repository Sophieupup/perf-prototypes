背景与目标

1.1 现状痛点

当前业绩运营依赖每周经营会议 PPT 通报业绩，数据从 OMS 运营管理系统导出后经 Excel 加工再呈现到 PPT， 数据链路长、时效低、无版本管理。当出现"上周说能签，本周说签不下来"这类预测漂移时， 无法回看历史预测内容，也无法定位变更原因，复盘困难。

1.2 系统目标

建立一套版本控制与快照机制，让业绩预测的每周变化可追溯、可对比、可复盘。具体达成：

  • 任何过去时点的预测均可回看（"上周怎么说的"一查便知）
  • 本周预测与上周不一致时，强制留下变更原因（不再"哑变更"）
  • 签约 / 收款预测报告自动按上周 / 本周 / 月初三组数据并列呈现
  • 所有锁定时点可一键导出完整历史快照
实施优先级
A 追溯（核心痛点，第一阶段必须达成）→ B 预测一致性管控、C 多时间点对比报告、D 完整快照引擎 同步落地。

功能总览

历史快照与时间线由 4 大功能模块构成：

#模块核心能力优先级
1 时间旅行与存档 双轨快照创建（自动定时 + 手动补录）· 交互式时间线 UI · 只读历史模式 P0
2 预测报告与数据获取 多点检索（上周 / 本周 / 月初 三组数据自由抽取）· 三列对比展示 P0
3 差异分析与变更日志 一致性检查触发变更理由必填 · 跨月延期重新承诺日期 · 商机暂停原因 · 推进记录聚合 P0
4 导出工具 任意锁定时点全量导出（PDF / Excel / 自动纪要 三种格式） P1

详细设计

3.1 快照创建 · 双轨制

系统通过自动定时与手动补录两个相互独立又互补的轨道创建快照，确保关键时点不缺失，特殊节点可灵活补充。

类型触发方式锁定时点
自动定时（主轨） 系统调度 每周五 12:00 + 每月初首个工作日 12:00
手动补录（副轨） admin 手动点击"锁定当前数据" 季度末 / 重要节点前 / 用户认为时机合适时

关键约束 · 每周仅保留 1 个值

周内若多次锁定，新锁定覆盖旧锁定，不累积版本号。导出时不会夹带半成品的中间值。

关键流程 · 解锁 / 修改 / 再锁定

已锁定的快照可由 admin 解锁 → 修改 → 重新锁定，不新建版本，保持历史链路干净。

W18 锁定（5/8 12:00 黄诗览）
  ↓ 5/9 admin 解锁
  ↓ 修改若干字段
  ↓ 5/9 11:30 重新锁定
  → W18 仍只是 1 个版本，覆盖更新
  → 不会出现 W18-a / W18-b 多版本

3.2 时间线 UI

提供横向时间线作为快照的导航与可视化入口。

  • 时间轴按月分段（JAN / FEB / MAR ...），左→右为时间正序
  • 已锁定日期 = 绿色实心点 · 可点击进入只读视图
  • 未锁定 / 未来日期 = 灰色虚线点 · 不可选
  • 当前 LIVE = 琥珀色脉动点 · 表示尚未持久化的活动状态
  • 点击任一绿点 → 主区域切换到该快照的只读视图

3.3 只读模式

选中过去快照后，整个 UI 立刻切换到只读状态，反映该时点的历史值。

  • 顶部条带提示"正在查看 W18 / 5/8 12:00 历史快照"，让用户清楚自己处于历史模式
  • 所有 input / button / select 灰显禁用，防止误改历史数据
  • 提供"← 返回 LIVE"按钮一键切回当前实时状态

3.4 多点数据检索

"签约和收款预测"报告支持从 3 个特定快照同时拉取数据：

检索点取数来源
上周上一个 W 锁定（如 W17）
本周当周 W 锁定 / 若未到锁定时点则取 LIVE
月初当月第 1 个锁定（如 M5）

报表中可同列展示上周 / 本周 / 月初三组数据，行级变化用 ↑↓ 箭头 + 差异值高亮。

3.5 一致性检查

当本周值与上周快照不同时，系统自动弹出 "变更理由" 必填项，未填写则提交被阻止。

触发字段范围采用最严格的 第 3 档：所有关键预测字段（含达成可能 + 预期时间 + 关键金额 三类）。具体见 §4 触发字段清单。

3.6 跨月延期

当 expected_sign_date（预计签单日期）或 planned_payment_date（预计回款日期）改到下个月（含跨季 / 跨年）时：

  • 仍触发 §3.5 的"变更理由"必填
  • 额外 必填 new_committed_date 重新承诺日期（不再"无限延期"）
  • 不做红边框 / Review 优先看 / 延期归类下拉等额外标记，保持表面简洁

3.7 商机暂停

当商机状态从 active 变更为 paused（暂停）时，系统自动触发以下行为：

字段行为
paused_at 暂停时间系统自动写入当前时间戳
paused_reason 暂停原因强制必填（同变更理由弹窗）
paused_by 暂停操作人系统自动记录当前用户

所有暂停信息聚合到推进记录中，详见 §3.8。

3.8 推进记录聚合

所有以下事件自动追加到 推进记录（进度账簿）：

  • §3.5 一致性检查的 变更理由
  • §3.6 跨月延期的 重新承诺日期
  • §3.7 商机暂停的 暂停原因

每条记录包含：

时间戳 + 操作人 + 字段名 + 原值 → 新值 + 原因 + 关联快照 ID
  • 不可删除，仅追加（审计要求）
  • 在商机详情"过程跟进 · 历史保留"区可见，按时间倒序展示

3.9 导出工具

支持基于任何历史"锁定"时间戳导出完整报告。三种格式适配不同场景：

格式适用场景内容
PDF给老板带变更原因 + 解读，适合阅读
Excel给运营明细表，适合手工分析
自动纪要给 Review 会议本周变化摘要，AI 生成

导出约束

  • 导出范围 = 该锁定时间点的全量数据 + 截至该时点的变更日志
  • 已被解锁后未重新锁定的快照 = 不可导出（避免半成品）
  • 每周 1 个快照 = 每周 1 份归档报告，导出无冗余（呼应 §3.1 每周 1 值约束）

触发字段清单

以下字段在快照对比中任一改变即触发"变更理由"必填：

字段类型字段 ID字段名称触发备注

数据模型

5.1 快照主表

CREATE TABLE opportunity_snapshots (
  snapshot_id        UUID PRIMARY KEY,
  snapshot_label     VARCHAR(32),    -- 'W18', 'M5', 'manual-20260507'
  snapshot_type      VARCHAR(16),    -- 'auto-week' | 'auto-month' | 'manual'
  locked_at          TIMESTAMP NOT NULL,
  locked_by          VARCHAR(64) NOT NULL,
  unlocked_at        TIMESTAMP NULL, -- 解锁时间，重新锁定后清空
  is_active_lock     BOOLEAN NOT NULL,
  data_payload       JSONB NOT NULL  -- 该时点全量商机/签单/回款数据
);

CREATE INDEX idx_snapshot_label ON opportunity_snapshots(snapshot_label);
CREATE INDEX idx_locked_at ON opportunity_snapshots(locked_at);

5.2 推进记录追加表

CREATE TABLE progress_log (
  log_id             UUID PRIMARY KEY,
  opportunity_id     VARCHAR(32) NOT NULL,
  event_type         VARCHAR(32) NOT NULL,
  -- 'change-reason' | 'cross-month-delay' | 'pause-reason'
  field_name         VARCHAR(64),
  old_value          TEXT,
  new_value          TEXT,
  reason             TEXT,
  new_committed_date DATE NULL,      -- 仅 cross-month-delay 用
  associated_snapshot_id UUID NULL,
  operator           VARCHAR(64) NOT NULL,
  created_at         TIMESTAMP NOT NULL DEFAULT now()
);

-- 仅允许 INSERT，禁止 UPDATE / DELETE（审计要求）
REVOKE UPDATE, DELETE ON progress_log FROM PUBLIC;
